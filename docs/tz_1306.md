# Мини-ТЗ: доработка CRM-прототипа до рабочего MVP

> **Для Claude Code.** Многое уже реализовано — правки точечные. Сначала изучи текущую кодовую базу и не переписывай то, что работает. Где в ТЗ написано «проверить» — это аудит существующего функционала на ошибки, а не реализация с нуля.

---

## P0 — Блокеры go-live

1. **Персист состояния при обновлении страницы.**
   Фильтры / перетаскивания / настройки не должны сбрасываться при reload.

2. **Smoke-тест флоу заведения клиента.**
   Прогнать полный путь заведения клиента из «Клиентов» и из «Вишлиста», убедиться, что компания попадает в единую базу и нет регрессий. (Баг с добавлением в «Клиентах» уже исправлен.)

## P1 — Функционал MVP

### Структура

- Переименовать узлы верхнего уровня:
  - «Административная» → **Управляющая компания (УК)**
  - «Производственное» → **Сервисные**
- Состав подразделений **не зашиваем** — его настраивают ответственные лица со стороны компании вручную в приложении.

### App Sidebar (навигация)

- **Акцентная подсветка** уже реализована для подразделений — распространить её также на App Sidebar.
- В App Sidebar **не видно, на какой странице сейчас пользователь** — добавить индикацию активного раздела (active state).

### Вишлист

- **Починить визуал drag-and-drop** — при перетаскивании нумерация / строки скачут некрасиво.
- **Проверить**, что есть возможность вручную убрать клиента в архив / подвал / потерянные (не дожидаясь авто-перевода через 3 года). Если нет — добавить.

### Целевые действия / KPI

> Сами целевые действия и их автофиксация актуальны и уже работают. Требуется только аналитический слой:

- **Отчёт** по целевым действиям.
- **Флоу выставления плана** (руководитель выставляет план по целевым действиям).
- **Данные на дашборде:**
  - **Руководитель** — сводка по менеджерам.
  - **Менеджер** — по своим целевым действиям в формате **план / факт**.

### Иерархия видимости (доступы)

- Уже реализована (менеджер видит свои карточки; руководитель — своё подразделение и ниже по иерархии). **Проверить, что нет ошибок.**

### Уведомления (email) — ТОЛЬКО настройка

> На этом этапе реализуем **исключительно конфиг**. Сам механизм отправки и триггеры — отдельной задачей позже. **Кроме настройки ничего не делать.**

- Отдельный **таб в глобальных настройках приложения** для конфигурации SMTP:
  - сервер (host), порт;
  - тип подключения / шифрование (none / SSL-TLS / STARTTLS);
  - логин, пароль;
- Использовать **nodemailer**.
- **Обязательно** — свитч включить / выключить уведомления.

## Definition of Done для MVP

- Состояние интерфейса не сбрасывается при перезагрузке страницы.
- Флоу заведения клиента проходится без регрессий из «Клиентов» и из «Вишлиста».
- Узлы структуры переименованы (УК, Сервисные).
- App Sidebar: акцентная подсветка применена, активная страница видна.
- Вишлист: визуал drag-and-drop починен; ручной перевод клиента в архив/подвал/потерянные присутствует.
- Целевые действия: отчёт работает; руководитель выставляет план; дашборды показывают сводку по менеджерам (руководитель) и план/факт по своим действиям (менеджер).
- Иерархия видимости проверена, ошибок нет.
- В глобальных настройках есть таб с SMTP-конфигом на nodemailer и свитчем вкл/выкл; сам механизм отправки пока не реализуется.
